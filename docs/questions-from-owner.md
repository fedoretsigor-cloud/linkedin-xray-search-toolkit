## Open Questions

### Q1. Requirement analysis output is too verbose

Owner concern:

- The user sees too much text after analyzing a vacancy URL.
- The `I understood...` block may not be useful.
- `Must have` and `Nice to have` should likely become short bullet points.

Current status: Open.

Initial direction:

- Replace long narrative with a compact `Search Brief` card.
- Keep role, location, seniority, and domain visible.
- Show must-have and nice-to-have as short bullets.
- Consider hiding/removing the `I understood...` text from the main UI.

### Q2. How are Role variants created?

Owner concern:

- Are role variants hardcoded or generated by the AI model from the vacancy?
- Why are there usually 3 variants?

Current status: Needs clarification in UI and docs.

Current implementation notes:

- Role variants from vacancy analysis are generated by the OpenAI requirement-analysis model through `src/requirement_agent.py`.
- There is no fixed count in the JSON schema.
- The prompt asks for useful search titles, and the model often returns 3.
- The UI currently applies up to 4 variants.
- Separate manual role presets exist in `static/role_presets.js`; those are hardcoded presets, but they are not the same as AI-generated requirement role variants.

### Q3. Why does Apply to Search Builder transform Role variants into Role titles?

Owner concern:

- After clicking `Apply to Search Builder`, role variants become search titles.
- The logic and reason are not obvious.
- Confirm whether this is hardcoded or AI-generated.

Current status: Open.

Current implementation notes:

- `Apply to Search Builder` copies AI-generated `brief.role_variants` into the editable Role Variants field.
- Search then treats `Role + Role Variants` as title inputs.
- This is deterministic UI logic, not another AI call.
- The variants themselves come from the requirement model.

### Q4. How are Must-have search anchors, Domain keywords, and Query groups created?

Owner concern:

- Need clarity on what creates these fields and what logic is used.

Current status: Open.

Current implementation notes:

- Requirement analysis returns `search_keywords`, `must_have_skills`, `nice_to_have_skills`, `domain`, and role data.
- `src/search_intent.py` currently builds Search Intent from these fields.
- The current Search Intent Builder is rule-based with a canonical terms list.
- It extracts terms like `Python`, `automotive`, `embedded`, `ECU`, `HiL`, `SIL`, `Git`, `JFrog`, etc.
- This was added to avoid long human requirement phrases becoming weak search query fragments.

### Q5. Why show many Tech Stacks if not all go to search?

Owner concern:

- The UI can show 8-10 tech stack values.
- It is unclear whether all are actually used, especially with a 20-candidate search limit.

Current status: Needs UI clarification.

Current implementation notes:

- Backend builds base queries across role titles, skill groups, locations, and sources.
- For a 20-candidate limit, each base query gets one provider query variant, but all base queries are still run.
- Final results are deduped and sliced to the requested candidate limit.
- The UI should make the difference clear between `query groups used for search` and `final result limit`.

### Q6. Location must be strict

Owner concern:

- Search should strictly respect location.
- Location is a hard condition.

Current status: Decision accepted, implementation in progress.

Current implementation notes:

- Location is currently included in the X-Ray query.
- There is no strict post-search location filter yet.
- Search providers can still return candidates outside the requested location because query matching is fuzzy.

Initial direction:

- Add strict location policy to Search Intent / search request. Started.
- Add post-filtering and/or strong score penalty when candidate location does not match. Started.
- Make UI explicitly show `Strict location: ON`. Started.

Decision:

- Location is a hard condition by default.
- Query text alone is not enough evidence.
- A result must show location evidence in indexed title/snippet/name to survive strict filtering.
- `Remote + Country/City` means "candidate must match that concrete country/city; remote is only work format".
- This applies to every country/city, not only Ukraine or France.
- When `remote` is paired with a concrete country/city, strict matching ignores `remote` and requires only concrete location evidence.

### Q7. Are Role and Role Variants all used in search?

Owner concern:

- Need clarity on whether `Role` and all `Role Variants` are used.

Current status: Needs UI explanation.

Current implementation notes:

- Yes, backend merges `role` and `titles` through `merge_role_titles`.
- Each title currently becomes a separate title input for query generation.

### Q8. Why not make Role primary and Role Variants OR alternatives?

Owner concern:

- Role should be the main value.
- Role variants should be alternatives under OR.

Current status: Open design decision.

Current implementation notes:

- Current query generation creates separate queries per title.
- It does not currently build one grouped title expression like `("Role" OR "Variant 1" OR "Variant 2")`.

Initial direction:

- Test grouped role queries.
- Balance recall, precision, and query length.

### Q9. Why not run one query with all required parameters as AND and optional as OR?

Owner concern:

- A single query could be more transparent:
- Root constraints with AND.
- Additional variants with OR.

Current status: Open design decision.

Current implementation notes:

- Current provider query string effectively combines groups with AND-like spacing and OR inside groups.
- The app splits into multiple base queries to avoid overly long queries and to improve recall.
- Need a clearer query planner that can decide between one strict query and multiple recall-expansion queries.

Initial direction:

- Add Query Planner modes:
- `strict`: one compact query with core AND constraints.
- `balanced`: several queries with role/domain/skill variations.
- `broad`: more recall-oriented expansion.

### Q10. Future chat-based sourcing copilot

Owner expectation:

- The system should eventually chat with the user to clarify who we are searching for.
- It should propose options, compose a search request together with the user, and ask for confirmation before search.
- The chat should use a strong OpenAI model with app context.
- Owner wants to define rules and limits for bot behavior.

Current status: Planned later, after workflow actions are stable.

Current implementation notes:

- Current code uses `OPENAI_MODEL` from environment, defaulting to `gpt-4o-mini`.
- A future chat layer can use a stronger model if configured.
- Bot behavior rules can and should be defined as product/system policy.

Initial direction:

- Build stable structured actions first.
- Then add chat as an interface over those actions.
- Store chat decisions as structured project state, not only free-form messages.
